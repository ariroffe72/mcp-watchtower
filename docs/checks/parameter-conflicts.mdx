---
title: "Parameter conflict detection across tools"
description: "MCP Watchtower warns when two tools use different names for the same parameter concept, such as ticker vs symbol, to prevent inconsistent LLM tool chaining."
---

When two tools accept what is logically the same input but name it differently — for example, one tool uses `ticker` and another uses `symbol` — a model composing those tools may pass values to the wrong parameter or fail to recognise that the same data can be reused. MCP Watchtower compares every pair of tools in your server against built-in synonym groups and flags pairs that use inconsistent names for the same concept.

## Finding details

| Field | Value |
| ----- | ----- |
| **Code** | `PARAMETER_CONFLICT` |
| **Severity** | `warning` |
| **`tool` field** | The name of the first tool in the conflicting pair |
| **`relatedTool` field** | The name of the second tool in the conflicting pair |

## Synonym groups

MCP Watchtower ships with the following synonym groups. If two tools each have a parameter from the same group but use different names, the pair is flagged.

| Concept | Recognised parameter names |
| ------- | -------------------------- |
| Stock identifier | `ticker`, `symbol`, `stock` |
| Record identifier | `id`, `identifier`, `key` |
| Search input | `query`, `search`, `q`, `term` |
| Result limit | `limit`, `max`, `count`, `size` |
| Pagination offset | `offset`, `skip`, `page` |
| Resource location | `url`, `uri`, `endpoint`, `href` |
| User reference | `user`, `username`, `user_id`, `userId` |
| Time reference | `date`, `timestamp`, `time`, `datetime` |

<Note>
  Each pair of tools is evaluated once per synonym group. If the same pair of tools conflicts in two different groups, you receive two separate findings.
</Note>

## Example

Suppose your server registers these two tools:

```json
[
  {
    "name": "get_quote",
    "description": "Returns the latest quote for a stock.",
    "inputSchema": {
      "type": "object",
      "properties": {
        "ticker": { "type": "string", "description": "The stock ticker, e.g. AAPL" }
      },
      "required": ["ticker"]
    }
  },
  {
    "name": "get_company_info",
    "description": "Returns company metadata.",
    "inputSchema": {
      "type": "object",
      "properties": {
        "symbol": { "type": "string", "description": "The stock symbol, e.g. AAPL" }
      },
      "required": ["symbol"]
    }
  }
]
```

Both `ticker` and `symbol` belong to the **Stock identifier** group. Because they differ, MCP Watchtower produces:

```json
{
  "severity": "warning",
  "code": "PARAMETER_CONFLICT",
  "tool": "get_quote",
  "relatedTool": "get_company_info",
  "message": "Parameter 'ticker' in 'get_quote' and 'symbol' in 'get_company_info' likely refer to the same concept — consider using consistent naming"
}
```

## How to fix it

Pick one name from the synonym group and apply it to every tool that accepts that concept.

<CodeGroup>

```json before
// get_quote
{ "ticker": { "type": "string" } }

// get_company_info
{ "symbol": { "type": "string" } }
```

```json after
// get_quote
{ "symbol": { "type": "string" } }

// get_company_info
{ "symbol": { "type": "string" } }
```

</CodeGroup>

<Tip>
  Choose the name that best matches the vocabulary your users and models already know. For financial tools, `symbol` is more widely recognised than `ticker`. For generic tools, prefer the shortest unambiguous option (for example, `id` over `identifier`).
</Tip>

<AccordionGroup>
  <Accordion title="Why does this matter for LLM tool use?">
    When a model chains tool calls, it often carries output from one tool into the input of the next. If `get_quote` returns a field called `ticker` but `get_company_info` expects `symbol`, the model has to infer that these are the same thing — and it may get it wrong, particularly in long chains or when working with many tools. Consistent parameter names remove that ambiguity entirely.
  </Accordion>
  <Accordion title="What if I intentionally use different names?">
    If two parameters genuinely refer to different concepts that happen to share a synonym-group name, you can suppress the finding by renaming one parameter to something outside the synonym group. For example, if your `page` parameter refers to a literal HTML page rather than a pagination offset, rename it to `html_page` or `page_content` to avoid the match.
  </Accordion>
</AccordionGroup>
