---
title: "Add MCP Watchtower to your CI pipeline"
description: "Use MCP Watchtower as a CI quality gate. Exit code 1 on critical findings works out-of-the-box with GitHub Actions and any other CI system."
---

MCP Watchtower exits with code `1` whenever it finds a critical issue, which means you can drop it into any CI pipeline as a blocking quality gate — no plugins, no configuration files, no extra setup. If a critical finding is introduced, the job fails and the merge is blocked.

## How the CI gate works

The exit code contract is simple:

- `0` — no critical findings; the job passes
- `1` — one or more critical findings; the job fails

Every CI system (GitHub Actions, GitLab CI, CircleCI, Buildkite, and others) treats a non-zero exit code as a failure by default, so no extra logic is needed on your end.

## GitHub Actions workflow

The example below runs MCP Watchtower against your compiled server on every pull request. The `--json` flag emits structured JSON output to stdout, which is easier to parse in downstream steps or upload as a build artifact.

<Steps>
  <Step title="Build your server">
    Make sure your MCP server is compiled and ready to start before the scan step runs.
  </Step>
  <Step title="Add the scan step">
    Add the `mcp-watchtower scan` step after your build step. No installation is needed — `npx` fetches the latest version on demand.
  </Step>
  <Step title="Review findings in the job log">
    If any critical findings are present, the step fails and prints the JSON report. Fix the issues and push again.
  </Step>
</Steps>

```yaml .github/workflows/mcp-lint.yml
name: MCP lint

on:
  pull_request:
  push:
    branches: [main]

jobs:
  mcp-lint:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Build server
        run: npm run build

      - name: Lint MCP tools
        run: npx mcp-watchtower scan --server "node dist/server.js" --json
```

<Tip>
Pin a specific version in production pipelines to avoid unexpected behavior from new releases: `npx mcp-watchtower@0.1.0 scan ...`
</Tip>

## Machine-readable output with --json

When you pass `--json`, MCP Watchtower writes the full report as a JSON object to stdout instead of the human-readable table. Stderr still receives connection progress messages.

```bash
npx mcp-watchtower scan --server "node dist/server.js" --json
```

The JSON report has the following shape:

```json
{
  "server": "server",
  "toolCount": 12,
  "findings": [
    {
      "severity": "critical",
      "code": "DUPLICATE_TOOL_NAME",
      "tool": "get_price",
      "message": "Tool name 'get_price' is defined 2 times in this server"
    }
  ],
  "passedAt": "2024-11-15T09:32:00.123Z"
}
```

You can pipe this output to `jq`, upload it as a build artifact, or POST it to a reporting endpoint.

## Manifest mode for faster CI

If spinning up a live server process in CI is slow or impractical, use `--manifest` to scan a pre-exported tools JSON file instead. No server process is started.

```bash
npx mcp-watchtower scan --manifest ./tools.json
```

Generate `tools.json` once during your build step (for example, by calling your server's `tools/list` handler directly), then reuse it for scanning. This approach is faster and avoids network or startup timeouts.

<Note>
Manifest mode accepts three JSON shapes: a raw array of tools, an object with a `tools` array (`{ "tools": [...] }`), or an MCP `result` envelope (`{ "result": { "tools": [...] } }`). See the [manifest mode guide](/guides/manifest-mode) for the full format reference.
</Note>

## Naming servers in the report

Use `--name` to give your server a meaningful label in the report output. This is useful when you scan multiple servers in the same pipeline.

```bash
npx mcp-watchtower scan --server "node dist/server.js" --name payments-api --json
```

## Platform mode in CI

If your CI pipeline validates servers that will be loaded together in an agent orchestrator, add `--platform` to elevate name collision findings from informational to critical. A collision that is harmless in an isolated server becomes a blocking problem when multiple servers share a tool namespace.

```bash
npx mcp-watchtower scan --server "node dist/server.js" --platform --json
```

See the [platform mode guide](/guides/platform-mode) for a full explanation.
