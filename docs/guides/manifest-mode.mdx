---
title: "Scan from a tools manifest file"
description: "Use --manifest to scan a pre-exported tools JSON file without starting a live server. Ideal for CI pipelines, offline environments, and snapshot testing."
---

Manifest mode lets you run MCP Watchtower against a static JSON file that describes your tools, instead of connecting to a running MCP server. No server process is started, no ports are opened, and no network traffic is generated. This makes it well suited for CI pipelines where starting a full server adds latency or requires environment setup that is difficult to replicate.

## Accepted JSON formats

MCP Watchtower accepts three JSON shapes for the tools manifest. You can use whichever format is most convenient to produce from your existing tooling.

<CodeGroup>

```json Raw array
[
  {
    "name": "get_stock_price",
    "description": "Get the current price of a stock",
    "inputSchema": {
      "type": "object",
      "properties": {
        "ticker": { "type": "string", "description": "Stock ticker symbol" }
      },
      "required": ["ticker"]
    }
  }
]
```

```json tools object
{
  "tools": [
    {
      "name": "get_stock_price",
      "description": "Get the current price of a stock",
      "inputSchema": {
        "type": "object",
        "properties": {
          "ticker": { "type": "string", "description": "Stock ticker symbol" }
        },
        "required": ["ticker"]
      }
    }
  ]
}
```

```json MCP result envelope
{
  "result": {
    "tools": [
      {
        "name": "get_stock_price",
        "description": "Get the current price of a stock",
        "inputSchema": {
          "type": "object",
          "properties": {
            "ticker": { "type": "string", "description": "Stock ticker symbol" }
          },
          "required": ["ticker"]
        }
      }
    ]
  }
}
```

</CodeGroup>

The `result` envelope format matches the raw response you get from the MCP `tools/list` JSON-RPC method, so you can save that response directly to disk and pass it to `--manifest` without any transformation.

## Tool schema reference

Each entry in the tools array must have `name` and `description`. The `inputSchema` field is optional but recommended — several checks inspect parameter names and types.

```typescript
interface ToolSchema {
  name: string
  description: string
  inputSchema?: {
    type: 'object'
    properties?: Record<string, { type: string; description?: string }>
    required?: string[]
  }
}
```

<Note>
Tools without `inputSchema` are still scanned. Checks that require parameter information (such as parameter conflict detection) will simply have less data to work with.
</Note>

## Using --manifest

Point `--manifest` at your JSON file:

```bash
npx mcp-watchtower scan --manifest ./tools.json
```

Combine it with `--json` for machine-readable output in CI:

```bash
npx mcp-watchtower scan --manifest ./tools.json --json
```

Add `--name` to give the server a meaningful label in the report:

```bash
npx mcp-watchtower scan --manifest ./tools.json --name payments-api
```

Add `--platform` if you are validating tools that will be loaded into a multi-server orchestrator:

```bash
npx mcp-watchtower scan --manifest ./tools.json --platform
```

## Stdin mode

If you do not want to write a file to disk, you can pipe the JSON directly to MCP Watchtower via stdin. When no `--server`, `--remote`, or `--manifest` flag is provided and stdin is not a TTY, MCP Watchtower reads from stdin automatically.

```bash
cat tools.json | npx mcp-watchtower scan
```

You can also pipe the output of another command directly:

```bash
curl -s "https://api.example.com/mcp/tools/list" | npx mcp-watchtower scan --name remote-export
```

<Tip>
Stdin mode accepts the same three JSON formats as `--manifest`. You do not need to normalize the output of your tooling before piping it.
</Tip>

## Generating a manifest from your server

A common CI pattern is to generate the manifest once at build time and cache it across steps. Here is an example using Node.js to export the tool list from your server:

<Steps>
  <Step title="Export the tool list in your build script">
    Add a script that starts your server, calls `tools/list`, writes the result to `tools.json`, and exits. The exact implementation depends on your MCP SDK.
  </Step>
  <Step title="Run the scan against the manifest">
    Use `--manifest` to scan the cached file. This avoids re-starting the server in a separate CI step.

    ```bash
    npx mcp-watchtower scan --manifest ./tools.json --json
    ```
  </Step>
  <Step title="Commit the manifest as a snapshot (optional)">
    Committing `tools.json` to your repository lets you diff it in pull requests and track how your tool surface area changes over time.
  </Step>
</Steps>

<Warning>
If you commit `tools.json`, make sure your CI regenerates it from source before scanning. Scanning a stale manifest can miss new issues introduced since the file was last updated.
</Warning>
