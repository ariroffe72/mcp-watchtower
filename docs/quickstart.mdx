---
title: "Get started with MCP Watchtower"
description: "Scan your first MCP server for tool conflicts and compatibility issues in under a minute using npx — no installation required, just Node.js 18+."
---

MCP Watchtower ships as an npm package and runs directly with `npx`, so you don't need to install anything before your first scan. All you need is Node.js 18 or later and access to an MCP server — local or remote.

<Steps>

<Step title="Run your first scan">

Pass the command that starts your MCP server to the `--server` flag. Watchtower will start the process, connect over stdio, fetch the tool list, and disconnect.

```bash
npx mcp-watchtower scan --server "python my_server.py"
```

When the scan starts, you'll see the banner printed to your terminal:

```text
  ╔══════════════════════════╗
  ║   ⊕  MCP-WATCHTOWER  ⊕  ║
  ║  shadow · params · names ║
  ╚══════════════════════════╝
```

After connecting, Watchtower prints a report like this:

```text
mcp-watchtower — my-server
12 tools scanned

⚠ WARNING [get_stock_price]  PARAMETER_CONFLICT
  Parameter 'ticker' in 'get_stock_price' and 'symbol' in 'get_quote' likely refer to the same concept — consider using consistent naming
  Related tool: get_quote

─────────────────────────────────────
0 critical  1 warning  0 info
```

</Step>

<Step title="Try different server runtimes">

The `--server` flag accepts any shell command that starts a valid MCP server process.

<CodeGroup>

```bash Python server
npx mcp-watchtower scan --server "python my_server.py"
```

```bash Node.js server
npx mcp-watchtower scan --server "node dist/server.js"
```

```bash Published package via uvx
npx mcp-watchtower scan --server "uvx my-published-server"
```

</CodeGroup>

</Step>

<Step title="Scan a remote MCP server">

If your MCP server is deployed behind an HTTP endpoint, use `--remote` with a bearer token.

```bash
npx mcp-watchtower scan --remote "https://api.example.com/mcp" --auth-token "$MCP_TOKEN"
```

<Tip>
  Store your token in an environment variable and reference it with `$MCP_TOKEN` to avoid exposing credentials in shell history.
</Tip>

</Step>

<Step title="Use a tools manifest for CI">

If spinning up a server isn't practical in your pipeline, you can pass a JSON file containing your tool definitions directly. This is useful when your build process already produces a tools manifest.

```bash
npx mcp-watchtower scan --manifest ./tools.json
```

The manifest can be a raw array of tools or an object with a `tools` key:

```json
[
  {
    "name": "get_stock_price",
    "description": "Returns the current price of a stock.",
    "inputSchema": { "type": "object", "properties": { "ticker": { "type": "string" } } }
  }
]
```

</Step>

</Steps>

## Useful flags

These flags work with any of the scan modes above.

| Flag | Description |
|------|-------------|
| `--json` | Output results as JSON instead of the human-readable report |
| `--platform` | Elevate name collision findings to critical (for orchestrators loading multiple servers) |
| `--max-tools <number>` | Set a custom tool count threshold (default: `20`) |
| `--name <name>` | Override the server name shown in the report |

```bash
# Machine-readable JSON output
npx mcp-watchtower scan --server "python my_server.py" --json

# Custom tool count threshold
npx mcp-watchtower scan --server "python my_server.py" --max-tools 15

# Override the server name in the report
npx mcp-watchtower scan --server "python my_server.py" --name my-api
```

## Exit codes

Watchtower uses exit codes so you can use it as a CI gate:

- **`0`** — no critical findings
- **`1`** — one or more critical findings

<Note>
  Warnings and info findings do not affect the exit code. Only `critical` severity findings trigger exit code `1`.
</Note>
