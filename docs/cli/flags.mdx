The `scan` command accepts eight flags that control how Watchtower connects to your MCP server, what name appears in the report, how results are formatted, and how severity levels are calculated. This page documents each flag individually with its short form, type, and constraints.

<Note>
  `--server`, `--remote`, and `--manifest` are mutually exclusive. You can only provide one input source per invocation. `--auth-token` is required whenever you use `--remote` and is silently ignored otherwise.
</Note>

## Input source flags

<ParamField path="--server" type="string">
  Short form: `-s`

  The shell command used to start your local MCP server process. Watchtower spawns this command as a child process, connects over stdio using the MCP protocol, calls `listTools`, and then terminates the process.

  Pass the full startup command as a single quoted string, including any arguments:

  ```bash
  npx mcp-watchtower scan --server "python my_server.py"
  npx mcp-watchtower scan --server "node dist/server.js --port 8080"
  npx mcp-watchtower scan -s "uvx my-published-server"
  ```

  The server name for the report is derived from the last segment of the command, with the file extension stripped. Use `--name` to override it.
</ParamField>

<ParamField path="--remote" type="string">
  Short form: `-r`

  The full URL of a remote MCP endpoint, for example `https://api.example.com/mcp`. Watchtower connects over HTTP using the MCP Streamable HTTP transport and calls `listTools`. The connection uses a bearer token supplied via `--auth-token`.

  ```bash
  npx mcp-watchtower scan \
    --remote "https://api.example.com/mcp" \
    --auth-token "$MCP_TOKEN"
  ```

  The server name defaults to the URL hostname. If the path ends in a segment other than `mcp`, that segment is appended to the hostname.
</ParamField>

<ParamField path="--auth-token" type="string" required>
  Short form: `-t`

  The bearer token sent in the `Authorization` header when using `--remote`. This flag is **required** when `--remote` is specified; omitting it causes an immediate error before any network request is attempted.

  ```bash
  npx mcp-watchtower scan \
    --remote "https://api.example.com/mcp" \
    --auth-token "$MCP_TOKEN"
  ```

  <Warning>
    If you use `--remote` without `--auth-token`, the command exits with: `Missing auth token. Use --auth-token when connecting with --remote.`
  </Warning>
</ParamField>

<ParamField path="--manifest" type="string">
  Short form: `-m`

  Path to a local JSON file that contains the tool definitions. No server is started or connected to — Watchtower reads the file and runs checks directly against it. This is the recommended approach for CI environments where spinning up a live server is impractical.

  ```bash
  npx mcp-watchtower scan --manifest ./tools.json
  npx mcp-watchtower scan -m /tmp/exported-tools.json
  ```

  The file must contain a raw array of tool objects, an object with a `tools` array, or an object with a `result.tools` array.
</ParamField>

## Report flags

<ParamField path="--name" type="string">
  Short form: `-n`

  A custom label used as the server name in the scan report. When omitted, the name is derived automatically from the startup command (`--server`), the URL hostname (`--remote`), or defaults to `unknown-server` for manifest and stdin modes.

  ```bash
  npx mcp-watchtower scan --server "node dist/server.js" --name my-api
  npx mcp-watchtower scan \
    --remote "https://api.example.com/mcp" \
    --auth-token "$TOKEN" \
    --name prod-api
  ```
</ParamField>

<ParamField path="--json" type="boolean">
  Short form: `-j`

  Emit the full scan report as a JSON object to stdout instead of the default human-readable format. The banner is suppressed when this flag is set.

  ```bash
  npx mcp-watchtower scan --server "node dist/server.js" --json
  ```

  The JSON output contains the server name, tool count, and an array of findings, each with `severity`, `code`, `message`, `tool`, and optionally `relatedTool` fields. This format is suitable for piping into other tools or storing as a CI artifact.
</ParamField>

## Severity and threshold flags

<ParamField path="--platform" type="boolean">
  Short form: `-p`

  Enables platform mode, which elevates name collision findings from informational severity to critical. Use this flag when you are building an agent orchestrator, an MCP registry, or any system that loads multiple servers simultaneously — environments where name collisions are genuinely dangerous because tool namespaces from different servers are merged into a single context.

  ```bash
  npx mcp-watchtower scan --server "node dist/server.js" --platform
  ```

  <Tip>
    If you are building a single-server MCP implementation rather than a multi-server platform, you can omit this flag. Name collision findings will still appear at a lower severity level.
  </Tip>
</ParamField>

<ParamField path="--max-tools" type="number" default="20">
  The maximum number of tools allowed before Watchtower emits a tool-count warning. Defaults to `20`. LLM routing accuracy tends to degrade above this threshold, so reduce the value if your models show signs of tool confusion at lower counts.

  ```bash
  npx mcp-watchtower scan --server "node dist/server.js" --max-tools 15
  npx mcp-watchtower scan --manifest ./tools.json --max-tools 10
  ```
</ParamField>
